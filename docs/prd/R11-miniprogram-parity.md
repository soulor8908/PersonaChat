# PRD: 小程序功能对齐 Web 端 (Round 11)

> **角色**: BA | **状态**: 回溯补齐 | **对应实现**: Round 11 (已交付)

---

## 一、背景与目标

### 1.1 背景

PersonaChat 存在两个客户端：Web 应用 (`apps/web/`) 和微信小程序 (`apps/miniprogram/`)。代码审查发现小程序端缺失或落后于 Web 端的以下能力：

- 首页缺少热门推荐模块、排序按钮、人格卡片统计信息和创建入口
- 首页加载和空状态使用简陋文本，缺乏骨架屏和图标化空状态
- 聊天页缺少工具调用状态指示器和停止生成按钮
- 个人设置页缺少 API Key 配置入口
- 全局缺少统一样式定义
- API 客户端未携带 `x-api-key` 头，导致有认证需求的接口不可用
- 历史页分组的标题错误地显示 `personaId` 而非 `personaName`

### 1.2 目标

实现小程序与 Web 端在核心功能上的对等，确保用户在两个端获得一致的体验。**不引入新功能**，只补齐差距。

### 1.3 约束

| 约束 | 说明 |
|------|------|
| AI-001 | PRD 应在编码前存在（本文档为回溯补齐） |
| AI-003 | 严格限定在"小程序对齐 Web 端"范围内 |
| AI-006 | 遇到歧义需标记 BLOCKING Q&A |

## 三、功能范围

### F1: 首页 — 热门推荐模块

**位置**: `index.wxml` + `index.wxss`

- 水平滚动卡片列表，每张卡片展示：头像（首字圆形渐变）、昵称、对话数 + 好评率
- 数据来源：`PersonaApi.listHot()` 已实现，补全 UI
- 无热门数据时整块隐藏（`wx:if` 守卫）

### F2: 首页 — 排序按钮

**位置**: `index.wxml` + `index.wxss`

- 三个排序选项：热门(popular) / 最新(recent) / 好评(rated)
- 点按时切换 `activeSort`，发请求重新加载列表
- 活动态高亮样式（紫色文字 + 浅紫背景）

### F3: 首页 — 人格卡片统计信息

**位置**: `index.wxml`

- 每张卡片底部展示：💬 messageCount / 👍 likeRate% / 🔧 toolCount（有条件显示）
- 工具数仅在 `item.tools.length > 0` 时渲染

### F4: 首页 — 创建入口按钮

**位置**: `index.wxml` + `index.wxss`

- 渐变色全宽按钮："＋ 创建人格"
- 点击跳转到创建页面

### F5: 首页 — 骨架屏 + 空状态

**位置**: `index.wxml` + `index.wxss`

- **骨架屏**: `loading` 为 true 且 `personas.length === 0` 时显示；包含旋转动画（`@keyframes spin`）+ "加载中..." 文字
- **空状态**: `personas.length === 0 && !loading` 时显示；包含 🔍 图标 + "暂无数据" 文字
- 旧版 `<view class="loading">加载中...</view>` 和 `<view class="empty">暂无数据</view>` 被移除

### F6: 聊天页 — 工具状态指示器 + 停止按钮

**位置**: `chat.wxml` + `chat.wxss` + `chat.js`

- **工具状态**: 当 `item.toolStatus` 非空时，在消息内容下方显示 `<view class="msg-tool-status">🔧 {{item.toolStatus}}...</view>`
- 样式：橙黄色 `#f0ad4e`，22rpx 字号
- **停止按钮**: 加载中（`loading` 为 true）时，发送按钮替换为红色"停止"按钮
- `onStop()` 方法调用 `this._streamTask.abort()` 中断流式请求，清理占位消息

### F7: 设置页 — API Key 配置

**位置**: `profile.wxml` + `profile.wxss` + `profile.js`

- 密码输入框（`password` 属性），值绑定 `apiKey`
- 保存按钮：`wx.setStorageSync('api_key', apiKey)` + Toast "已保存"
- 保存后按钮文字暂时变为 "已保存 ✓"（2 秒后恢复）
- 提示文字："用于访问 PersonaChat API 的认证密钥"

### F8: 全局 — 请求头携带 API Key

**位置**: `api/client.js`

- 每次请求从 `wx.getStorageSync('api_key')` 读取 Key
- 非空时附加到 `header['x-api-key']`

### F9: 历史页 — 分组标题修复

**位置**: `history.js`

- 分组 key 使用 `chat.personaName || key` 替代 `personaId`
- 向后兼容：当 `personaName` 为 `undefined` 时降级为 key

## 五、每个功能的 Given/When/Then

### F1: 热门推荐模块

#### 正常路径

| GIVEN | WHEN | THEN |
|-------|------|------|
| 首页加载完成，`hotPersonas` 有 >= 1 条数据 | 页面渲染 | 热门推荐区域可见，标题显示"🔥 热门推荐"，卡片水平滚动 |
| 热门卡片渲染完成 | 用户向左滑动 | 后续卡片可见，滚动流畅无卡顿 |
| 用户点击热门卡片 | 触发 `bindtap` | 跳转至聊天页，传入该人格 ID |

#### 边界条件

| GIVEN | WHEN | THEN |
|-------|------|------|
| `hotPersonas` 为空数组 | 页面渲染 | 整个热门区域隐藏（`wx:if` 判断），不占用布局空间 |
| 热门数据正在加载中 | 页面渲染 | 不阻塞首页整体渲染，热门区域在数据到达后才显示 |
| 人格名称为空字符串 | 卡片渲染 | 头像取首字母为空，显示""空字符；名称区域为空文本 |
| 热门数据量极大（> 50 条） | 渲染 | 横向滚动正常，无白屏或性能问题 |

#### 错误路径

| GIVEN | WHEN | THEN |
|-------|------|------|
| `PersonaApi.listHot()` 请求失败（网络异常/500） | 页面加载 | 热门区域不显示，不影响其他模块；控制台无报错 |
| `hotPersonas` 返回异常格式（非数组） | 赋值 | `res.data || []` 兜底为空数组，热门区域隐藏 |

### F2: 排序按钮

#### 正常路径

| GIVEN | WHEN | THEN |
|-------|------|------|
| 首页渲染完毕，`sortOptions` 包含 popular/recent/rated | 页面展示 | 三个排序按钮水平排列，默认 popular 为 active 高亮态 |
| 用户点击 "最新" 按钮 | 触发 `onSortTap` | `activeSort` 切换为 `recent`，按钮样式变为高亮，重新加载列表 |
| 列表重新加载完成 | 接口返回 | 人格列表按创建时间倒序排列 |

#### 边界条件

| GIVEN | WHEN | THEN |
|-------|------|------|
| 用户快速连续点击不同排序按钮 | 高频触发 | 每次切换都发起独立请求，最后设置的排序生效；避免竞态条件（上次请求返回覆盖最新结果） |
| 当前排序下列表为空 | 加载完成 | 显示空状态图标 🔍 + "暂无数据" |

#### 错误路径

| GIVEN | WHEN | THEN |
|-------|------|------|
| 排序请求失败 | 网络错误 | 列表维持之前的数据，不刷新；静默处理异常 |

### F3: 人格卡片统计信息

#### 正常路径

| GIVEN | WHEN | THEN |
|-------|------|------|
| 人格卡片渲染 | 页面展示 | 卡片内显示 💬 messageCount + 👍 likeRate% |
| `item.tools` 存在且长度 > 0 | 卡片渲染 | 额外显示 🔧 N 工具 |
| messageCount 为 0 | 渲染 | 显示 💬 0 |
| likeRate 为小数（如 0.85） | 渲染 | 显示 👍 85%（`Math.round(item.likeRate * 100)`） |

#### 边界条件

| GIVEN | WHEN | THEN |
|-------|------|------|
| `item.messageCount` 为 `undefined` | 渲染 | 显示 💬 undefined（需确保 API 始终返回该字段或前端做兜底） |
| `item.likeRate` 为 `null` | 渲染 | `Math.round(null * 100)` 结果为 0，显示 👍 0% |
| `item.tools` 为 `undefined` | 渲染 | `wx:if` 条件为 false，不显示工具数条目 |
| messageCount 值极大（如 99999） | 渲染 | 正常显示数字，无格式化/截断需求 |

#### 错误路径

| GIVEN | WHEN | THEN |
|-------|------|------|
| `item` 对象为 `null` | 遍历渲染 | 小程序框架静默跳过，控制台报 warning（需确保 API 返回无脏数据） |

### F4: 创建入口按钮

#### 正常路径

| GIVEN | WHEN | THEN |
|-------|------|------|
| 首页渲染 | 页面展示 | 顶部可见渐变紫色全宽按钮 "＋ 创建人格" |
| 用户点击按钮 | 触发 `onCreateTap` | 跳转至创建页（`/pages/create/create`） |

#### 边界条件

| GIVEN | WHEN | THEN |
|-------|------|------|
| 按钮文本过长 | 布局 | 按钮高度可自适应，无溢出 |
| 用户从创建页返回 | 返回首页 | 按钮正常显示，无状态残留 |

#### 错误路径

| GIVEN | WHEN | THEN |
|-------|------|------|
| 创建页面路径不存在 | 点击按钮 | 跳转失败，微信开发者工具报 navigateTo 错误（属于项目配置问题，非本功能缺陷） |

### F5: 骨架屏 + 空状态

#### 正常路径

| GIVEN | WHEN | THEN |
|-------|------|------|
| 页面加载中，`loading` 为 true 且 `personas.length === 0` | 渲染 | 显示骨架屏：旋转 spinner + "加载中..." |
| 数据加载完成，`personas.length === 0` | 渲染 | 显示空状态：🔍 图标 + "暂无数据" |
| 数据加载完成，`personas.length > 0` | 渲染 | 骨架屏和空状态均隐藏，显示人格列表 |

#### 边界条件

| GIVEN | WHEN | THEN |
|-------|------|------|
| `loading` 为 true 但 `personas.length > 0`（增量刷新场景） | 渲染 | 不显示骨架屏（因为已有数据），保留已有列表 |
| 数据从 0 变为 > 0 | 过渡 | 骨架屏直接消失，切换为人格列表；无动画过渡 |
| 首次加载极快（< 100ms） | 渲染 | 骨架屏闪一下后消失，不会造成视觉不适（可接受） |

#### 错误路径

| GIVEN | WHEN | THEN |
|-------|------|------|
| 加载失败，`loading` 变为 false | 错误处理 | 显示空状态（🔍 + "暂无数据"），而非骨架屏；静默处理异常 |

### F6: 工具状态指示器 + 停止按钮

#### 正常路径

| GIVEN | WHEN | THEN |
|-------|------|------|
| LLM 调用工具且前端收到 `tool_start` 事件 | 消息渲染 | 在助手消息下方显示 "🔧 toolName..." 状态文字，颜色 `#f0ad4e` |
| 工具执行完成，收到 `tool_end` 事件 | 状态更新 | 工具状态文字消失，消息内容更新为最终回答 |
| 助手消息正在流式生成 | 页面渲染 | 发送按钮变为红色 "停止" 按钮 |
| 用户点击停止按钮 | 触发 `onStop` | 调用 `this._streamTask.abort()` 中断流式请求，清理流式占位消息，`loading` 设为 false |

#### 边界条件

| GIVEN | WHEN | THEN |
|-------|------|------|
| 连续多次工具调用（tool_use 循环 > 1 轮） | 状态变化 | 每次工具调用都正确更新 toolStatus，前一次状态被后一次覆盖 |
| 用户在网络延迟时点停止 | 正在建立连接 | 如果 `this._streamTask` 有 abort 方法则调用，无则静默跳过 |
| 停止后已有内容 | 中断完成 | 已渲染的文本内容保留在界面上，不清除 |
| 停止时助手消息仅有 toolStatus 无内容 | 清理 | 该消息被标记为 `null` 并通过 `filter(Boolean)` 过滤掉 |

#### 错误路径

| GIVEN | WHEN | THEN |
|-------|------|------|
| `this._streamTask` 为 null/undefined | 点击停止 | `onStop` 中 `if` 判断为 false，不执行 abort；`loading` 设为 false，消息清理执行 |
| `this._streamTask.abort()` 抛出异常 | 中断 | `onStop` 未包裹 try-catch，异常向上冒泡（需检查是否需要容错） |
| 工具调用阶段 SSE 连接断开 | 流中断 | 前端应能检测到断开并将 `loading` 设为 false，恢复 UI |

### F7: API Key 配置

#### 正常路径

| GIVEN | WHEN | THEN |
|-------|------|------|
| 用户进入设置页 | 页面加载 `onLoad` | 输入框预填 `wx.getStorageSync('api_key')` 的值 |
| 用户输入 API Key 并点击保存 | 触发 `onSaveApiKey` | `wx.setStorageSync('api_key', value)` 保存，Toast "已保存"，按钮文字变为 "已保存 ✓" |
| 2 秒后 | 计时器触发 | 按钮文字恢复为 "保存" |

#### 边界条件

| GIVEN | WHEN | THEN |
|-------|------|------|
| 从未保存过 API Key | 页面加载 | 输入框显示空字符串 `''` |
| 用户输入空格作为 API Key | 保存 | 空格被原样保存到 storage（无 trim） |
| API Key 长度极长（如 1024 字符） | 输入 | 输入框应可容纳，需确认小程序 input 组件无长度限制问题 |
| 用户连续快速点击保存 | 高频触发 | 每次保存覆盖前值；Toast 可能叠加，setTimeout 的 2 秒恢复逻辑不幂等 |

#### 错误路径

| GIVEN | WHEN | THEN |
|-------|------|------|
| `wx.setStorageSync` 存储空间不足 | 保存 | 抛出异常，当前未捕获（需考虑是否加 try-catch） |
| 用户清除小程序缓存 | 下次进入 | `api_key` 不存在，输入框为空，无需特殊处理 |

### F8: 请求头携带 API Key

#### 正常路径

| GIVEN | WHEN | THEN |
|-------|------|------|
| `wx.getStorageSync('api_key')` 返回非空值 | 发起请求 | 自动附加 `x-api-key` 头到所有 API 请求 |
| 后端验证通过 | 请求到达 | 正常返回数据 |

#### 边界条件

| GIVEN | WHEN | THEN |
|-------|------|------|
| 未保存 API Key（storage 值为空字符串） | 发起请求 | `if (apiKey)` 为 false，不附加头，行为向后兼容 |
| 用户保存 Key 后删除重新输入 | 下次请求 | 新 Key 自动生效 |

#### 错误路径

| GIVEN | WHEN | THEN |
|-------|------|------|
| 保存的 API Key 已过期/无效 | 请求到达后端 | 后端返回 401/403，前端按现有错误处理逻辑显示错误信息 |
| `wx.getStorageSync` 抛出异常（极少发生） | 发起请求 | 异常冒泡到外层，未处理（极低概率，可接受） |

### F9: 历史页分组标题修复

#### 正常路径

| GIVEN | WHEN | THEN |
|-------|------|------|
| 历史聊天记录中的 `personaName` 字段有效 | 分组渲染 | 分组标题显示为 `personaName` 而非 `personaId` |
| 用户查看不同人格的历史 | 分组展示 | 每组标题正确显示对应的可读名称 |

#### 边界条件

| GIVEN | WHEN | THEN |
|-------|------|------|
| `personaName` 为 `undefined` 或 `null` | 分组渲染 | 降级显示 `personaId`（`chat.personaName || key`） |
| `personaName` 为空字符串 `''` | 分组渲染 | 空字符串为 falsy，显示 `personaId` |
| 同人格名称为中文/特殊字符 | 渲染 | 正常显示，无编码问题 |

#### 错误路径

| GIVEN | WHEN | THEN |
|-------|------|------|
| API 返回的 `personaName` 包含 HTML 标签 | 渲染 | 直接渲染为文本（小程序默认不解析 HTML，安全） |

## 七、BLOCKING Q&A

> 以下为 BA 在编写 PRD 过程中发现的歧义或缺失信息，需要相关方确认后方可推进后续轮次。

### Q1: 热门推荐的排序 / 选取逻辑是什么？

**问题**: `PersonaApi.listHot()` 返回的"热门"数据是按什么标准选取的？是按 messageCount 降序、likeRate 降序，还是由某个算法综合评分？此信息影响验收时的数据验证标准。

**建议确认方**: 后端开发 / PM

### Q2: API Key 的安全性和过期策略？

**问题**: API Key 通过 `wx.setStorageSync` 存储在本地，但：
- 小程序端是否需要在保存前对 Key 做格式校验（如 JWT 三段式验证）？
- 后端是否有 Key 过期机制？过期后前端如何处理（静默重定向到设置页？弹出提示？）
- Key 是否支持多环境（dev/prod）？

**建议确认方**: 后端开发 / 安全负责人

### Q3: 工具状态指示器在流式中断或异常时的清理逻辑？

**问题**: 当前 `onStop()` 通过 `filter(Boolean)` 清理流式占位消息。但以下场景未覆盖：
- 工具调用中（tool_start 已发送但 tool_end 未到）用户按下停止 → 应显示什么样的 UI？是否保留"🔧 toolName..."的状态文字？
- SSE 连接意外断开（非用户主动停止）→ loading 状态和 toolStatus 如何清理？
- 后端工具调用超时（如 web_search 超过 10 秒）→ 前端是否应有超时兜底？

**建议确认方**: 后端开发 / 前端开发

### Q4: 骨架屏与空状态在排序切换时的行为？

**问题**: 当用户在不同排序间切换时，当前实现逻辑会：
1. 设 `loading = true`
2. 发起请求
3. 请求完成更新列表

在步骤 1-2 之间，如果已有 `personas.length > 0`，骨架屏不会显示（正确行为）。但如果用户切换到某个排序后列表为空（返回 0 条数据），步骤 3 完成后显示的是空状态。这中间是否需要显示骨架屏（因为 `loading` 为 true 但已有数据所以不显示）？快速排序切换时是否有视觉跳动？

**建议确认方**: 前端开发 / UX
